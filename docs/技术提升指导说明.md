## P0-1: CSRF 防护 — UA 检测绕过修复

### 问题
旧版 `CsrfFilter.isSsrRequest()` 通过检查 `User-Agent` 是否包含 `"nuxt"` 或 `"axios"` 来识别服务端发起的请求。攻击者只需在浏览器中设置 `User-Agent: axios` 就能绕过所有 POST/PUT/DELETE 的 CSRF 校验。

### 最佳实践
**双重提交 Cookie 模式的核心原则**：CSRF 攻击的本质是攻击者无法读取跨域 Cookie，因此只要保证请求头中的 Token 与 Cookie 中的 Token 一致，就能防御 CSRF。对于服务器间调用场景，应使用 **内网 IP 白名单**，而非客户端可伪造的 UA 或请求头。

```
错误：if (userAgent.contains("axios")) → 跳过 CSRF
正确：if (isInternalNetworkRequest(request)) → 跳过 CSRF
```

### 延伸阅读
- OWASP CSRF Prevention Cheat Sheet: Double Submit Cookie Pattern
- 服务器间 → API 通信应走内网或使用预共享密钥，不应依赖 UA 判断

## P0-2: Admin Token 自动刷新

### 问题
Admin 管理后台只存储 `accessToken`，不存储 `refreshToken`。Token 过期后直接跳登录页，管理员每 30 分钟需重新登录。

### 技术方案 — 全局刷新锁 + 请求队列

```
登录 → 存储 accessToken + refreshToken + expiresAt
                ↓
请求前 → Token 即将过期（剩余 < 5min）→ 主动刷新
                ↓
401 响应 → 检查是否有 refreshToken
         → 有：发起刷新请求，其他并发 401 加入等待队列
         → 刷新成功：重试所有挂起请求
         → 刷新失败：清除 Token，跳转登录
```

### 关键设计决策：避免循环依赖

Admin 的 `request.ts` 使用 `storage` 直接读写 Token，而非通过 Pinia `useUserStore()`：

```
❌ 循环依赖：request.ts → userStore → authApi → request.ts
✅ 单向依赖：request.ts → storage（底层工具）→ 无反向引用
```

### 最佳实践
- **静默刷新**：用户无感知，只有刷新失败才弹提示
- **全局单锁**：`globalIsRefreshing` + `pendingRequests[]` 防止并发刷新
- **阈值提前量**：5 分钟提前量，避免 Token 在传输途中过期

## P0-3: Client 统一 API 调用层

### 问题
`pages/index.vue` 中多处使用 Nitro `$fetch` 直接请求后端，绕过了 `useApi()` 提供的 Token 自动刷新、CSRF 注入等中间件能力。

### 技术方案
所有 API 调用统一通过 `useApi()` → Axios 实例 → 拦截器链：

```
useApi().get(url, params)
    → 请求拦截器：注入 Token + CSRF + 统一错误处理
    → 服务端处理
    → 响应拦截器：版本号检查 + 401 刷新 + 错误映射
```

### 最佳实践
- **`useAsyncData` 中用 `$fetch`**：因为 `$fetch` 是项目统一封装的 HTTP 工具，外部模块和 SSR 场景兼容性更好
- **客户端路径**：所有用户交互（刷新、翻页、点赞等）必须走 `useApi()`
- **数据路径**：`useApi()` 返回 `AxiosResponse<ApiResponse<T>>`，数据在 `res.data.data`

## P0-4: SQL 注入防护 — MyBatis-Plus 分页

### 问题
```java
// ❌ SQL 字符串拼接，代码异味
wrapper.last("LIMIT " + (page - 1) * pageSize + ", " + pageSize);
```
虽然 `page/pageSize` 是 `Integer` 类型不会被注入，但 `.last()` 是 MyBatis-Plus 的"最后追加 SQL"方法，字符串拼接在任何情况下都是安全隐患。

### 修复
```java
// ✅ MyBatis-Plus 分页，SQL 由框架安全生成
Page<Article> articlePage = new Page<>(page, pageSize);
articleMapper.selectPage(articlePage, wrapper).getRecords();
```

### 最佳实践
- **永不使用 `.last()` 拼接用户输入**，哪怕是数字类型
- 分页统一用 `Page<T>` + `selectPage()` 或 `IPage<T>` 接口
- `LambdaQueryWrapper` 已安全，但 `.last()` 绕过了安全保护

## P0-5: 认证防御性编程

### 问题
`SearchController.clearHistory()` 调用 `securityUtil.getCurrentUserId()` 但方法上无认证注解。虽然 `SecurityConfig.anyRequest().authenticated()` 已拦截未认证请求，但缺少显式保护。

### 修复
```java
@DeleteMapping("/history")
@PreAuthorize("isAuthenticated()")  // 双重保障
public R<Void> clearHistory() { ... }
```

### 最佳实践
- **纵深防御**：Filter 层（`SecurityConfig`）+ 方法层（`@PreAuthorize`）双重保护
- 任何访问 `getCurrentUserId()` 的方法都应标注认证要求
- 即使 `SecurityConfig` 已配置，方法级注解提供自文档化

## P1-1: 响应式断点规范

### 问题
Tailwind 配置只有 5 档断点（480/768/1024/1200/1920），缺少 xs 档（<480px）。小屏手机（320-375px）的适配依赖默认样式，可能不精确。

### 修复
```ts
screens: {
  xs: '375px',   // 新增 iPhone SE/6/7/8 等小屏设备断点
  sm: '480px',
  md: '768px',
  lg: '1024px',
  xl: '1200px',
  '2xl': '1920px',
}
```

### 最佳实践
- 断点应采用"移动优先"策略：默认样式 = 最小屏幕，断点 = 逐步增强
- 375px 是 iPhone 6/7/8/SE 的逻辑宽度，覆盖主流小屏
- 断点定义为调试点，不应为每个断点强制写样式

## P1-8: useBreakpoints 响应式值陷阱

### 问题
```ts
// ❌ current 是 ComputedRef<string>，Array.isArray 永远返回 false
const current = breakpoints.current()
// ✅ 必须取 .value 才能拿到实际字符串值
const current = breakpoints.current().value
```

### 根因
`@vueuse/core` 的 `useBreakpoints().current()` 返回 `ComputedRef<string>`，需要通过 `.value` 取值。在 JS 中 `Array.isArray(ComputedRef)` 永远返回 `false`，导致 `isTablet` 计算始终为 `false`。

### 最佳实践
- VueUse v10+ 返回 `ComputedRef`，需要对结果取 `.value`
- 使用 `unref()` 更安全：`unref(breakpoints.current())`
- 写单元测试验证断点判断逻辑

## P1-7: 搜索防抖标准化

### 问题
手动 `setTimeout` + `clearTimeout` 实现搜索建议防抖，存在闭包陷阱和清理不完整风险。

### 修复
```ts
// ❌ 手动 setTimeout
let timer: NodeJS.Timeout
const onInput = () => { clearTimeout(timer); timer = setTimeout(fetch, 300) }

// ✅ VueUse useDebounceFn
const debouncedFetch = useDebounceFn(fetchSuggestions, 300)
onUnmounted(() => debouncedFetch.cancel())
```

### 最佳实践
- 使用 VueUse 标准工具，避免手动防抖的闭包和清理问题
- `onUnmounted` 中 `.cancel()` 确保组件卸载时清理
- 防抖 300ms：平衡响应速度与请求频率

## P1-3: Redis 降级保护

### 问题
多个 Service 中 `stringRedisTemplate` 操作没有 try-catch 保护。Redis 连接断开或超时时直接抛异常，导致业务不可用。

### 修复策略

| 场景 | 降级策略 |
|------|---------|
| 登录失败计数 | Redis 不可用时跳过计数（不阻止登录） |
| Token 黑名单 | Redis 不可用时跳过黑名单检查 |
| 缓存读写 | Redis 不可用时直接查数据库 |
| 时间线 ZSet | Redis 不可用时降级到数据库直查 |

### 最佳实践
- **缓存是优化，不是依赖** — Redis 不可用时业务必须能降级运行
- try-catch 覆盖所有 Redis 调用点（Auth: 8处, Feed: 17处）
- 统一 log.warn 级别，监控可据此触发告警

## P1-4: OpenSearch 自动降级

### 问题
OpenSearch 连接断开时，搜索直接返回 503 "搜索服务暂时不可用"，用户无法搜索。项目已有 `FallbackService.getSearchFallback()`（MySQL LIKE 查询）但未被自动调用。

### 修复
```java
try {
    // OpenSearch 查询
    return searchResponse;
} catch (Exception e) {
    log.warn("OpenSearch 不可用，降级到 MySQL 搜索: {}", e.getMessage());
    return fallbackService.getSearchFallback(keyword, page, pageSize);
}
```

### 最佳实践
- **降级优先于报错** — 用户看到降级结果（可能不够准）好过看到 503
- 双重保护：Sentinel 熔断 + 代码级 try-catch
- 降级日志应标注为 WARN 级别以便监控

## P1-5: WebSocket 客户端守卫

### 问题
```ts
// ❌ 模块顶层直接访问 window，构建时或 SSR 时会立即崩溃
const baseUrl = window.location.origin.replace('http', 'ws')

// ✅ 加客户端守卫（typeof window !== 'undefined'）
const baseUrl = import.meta.client ? window.location.origin.replace('http', 'ws') : ''
```

### 最佳实践
- 任何使用 `window`/`document`/`localStorage` 的代码必须加 `import.meta.client` 守卫
- SPA 模式下构建时不应直接调用 `window`，需要运行时守卫
- WebSocket URL 通过 `import.meta.env.VITE_WS_BASE` 注入，缺省时回退到当前 origin

## 技术债务 — 后续建议

以下 P2 问题未在本次修复中处理，建议后续迭代关注：

1. **共享类型包**：Client/Admin/Server 各定义 `ApiResponse<T>`, `User`, `Article` 等类型，应抽取为 `packages/shared-types/`
2. **深色模式**：Tailwind `darkMode: 'class'` 已配置但无 `.dark:` 样式
3. **错误码枚举**：前后端错误信息用硬编码字符串，应统一 `ErrorCode` 枚举
4. **WebSocket 认证**：当前 URL 传 Token，应改为连接后握手认证
5. **图片优化**：使用 `<picture>` + WebP/AVIF 来源，懒加载 + srcset 适配多分辨率
6. **性能监控**：采集 FCP/LCP/CLS 等 Web Vitals 指标
