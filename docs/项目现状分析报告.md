#### 问题 3：JWT Secret 不安全
**影响范围**：认证安全  
**问题描述**：
- `application.yml` 第 141 行：默认 JWT Access Secret 不符合 256 位要求
- `application.yml` 第 143 行：默认 JWT Refresh Secret 不符合 256 位要求

**当前配置**：
```yaml
jwt:
  access-secret: ${JWT_ACCESS_SECRET:-zhixun-access-secret-key-2024-must-be-at-least-256-bits-long}
  refresh-secret: ${JWT_REFRESH_SECRET:-zhixun-refresh-secret-key-2024-must-be-at-least-256-bits-long}
```

**问题分析**：
1. 默认值长度可能不足 256 位（32 字节）
2. 默认值固定，所有部署实例共享相同密钥
3. 未强制要求生产环境配置环境变量

**修复建议**：
1. 生成强随机密钥（至少 32 字节 / 256 位）
   ```bash
   openssl rand -base64 32  # 生成 Base64 编码的 32 字节随机密钥
   ```
2. 在 `.env.example` 中添加说明，强制要求生产环境配置 `JWT_ACCESS_SECRET` 和 `JWT_REFRESH_SECRET`
3. 在应用启动时校验密钥长度，如果不足 32 字节则拒绝启动

---

#### 问题 4：AES Secret Key 不安全
**影响范围**：数据加密安全  
**问题描述**：
- `application.yml` 第 155 行：默认 AES Key 不符合 AES-256 要求（需要 32 字节）

**当前配置**：
```yaml
aes:
  secret-key: ${AES_SECRET_KEY:-zhixun-aes-secret-32bytes-key!!}
```

**问题分析**：
1. 默认值 `"zhixun-aes-secret-32bytes-key!!"` 长度 = 33 字节，但可能包含非随机字符
2. AES-256 要求 **精确 32 字节**（不是 32 字符的任意编码）
3. 应该使用加密安全的随机数生成器生成密钥

**修复建议**：
1. 生成强随机 AES-256 密钥：
   ```bash
   openssl rand -base64 32  # 生成 Base64 编码的 32 字节随机密钥
   ```
2. 在 `.env.example` 中添加说明，强制要求生产环境配置 `AES_SECRET_KEY`
3. 在 `AesUtil.java` 中添加密钥长度校验

#### 问题 6：MyBatis-Plus 日志输出到控制台
**影响范围**：生产环境性能  
**问题描述**：
- `application.yml` 第 130 行：`log-impl: org.apache.ibatis.logging.stdout.StdOutImpl`
- `application-dev.yml` 第 30 行：同样配置了控制台日志

**问题分析**：
1. 生产环境不应该将 SQL 日志输出到控制台，会影响性能
2. 控制台日志无法持久化，不便于问题排查

**修复建议**：
1. 生产环境配置改为日志文件输出：
   ```yaml
   mybatis-plus:
     configuration:
       log-impl: org.apache.ibatis.logging.slf4j.Slf4jImpl
   ```
2. 在 `logback-spring.xml` 中配置 MyBatis 日志输出到文件
3. 开发环境可以保留控制台日志，但生产环境必须关闭

## 四、优化方案建议

### 4.1 结构优化

#### 4.1.1 后端结构优化
| 优化项 | 当前状态 | 优化方案 | 优先级 |
|--------|----------|----------|--------|
| **统一异常处理** | 已配置 GlobalExceptionHandler | 检查是否所有异常都被正确处理，添加更多业务异常类型 | P1 |
| **参数校验** | 使用了 @Valid 注解 | 检查是否所有入参都做了校验，添加自定义校验注解 | P1 |
| **日志记录** | 使用了 Slf4j | 添加日志脱敏（如手机号、邮箱），配置日志滚动策略 | P1 |
